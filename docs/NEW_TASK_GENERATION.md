# Generating a New TAALI Task

## Overview

New tasks are generated by Claude using three inputs provided by a human operator:

1. The job specification (JD) for the target role
2. `docs/TAALI_TASK_SCORING_FRAMEWORK.md` — the authoritative design and scoring guide
3. Any additional role context or must-cover / must-not-cover constraints

The output is a single `.json` file placed in `backend/tasks/` that meets the full spec contract defined in the framework.

---

## Prompt to use

Paste the following into Claude with the three inputs attached or quoted inline:

```
I need you to generate a new TAALI assessment task JSON for the following role.

You must follow the TAALI_TASK_SCORING_FRAMEWORK.md (attached) in full.
Pay particular attention to:
- Part 2 (Signal Map): every task design decision must map to a scoring engine signal
- Part 4 (Repository Design): diagnostics must contain specific IDs, error text, and .py function names
- Part 5 (Rubric Engineering): 5 dimensions, unequal weights, observable criteria
- Part 7 (Quality Gates): run the signal audit checklist before producing the final JSON

**Role:** [paste exact job title]

**Job specification:**
[paste full JD text]

**Additional constraints (optional):**
- Must cover: [any specific requirements to prioritise]
- Must not cover: [anything explicitly out of scope]
- Business scenario suggestion: [optional — e.g. "compliance audit failure", "API gateway outage"]

**Output format:**
A single valid JSON object matching the spec contract in the framework.
All repo files must be included inline under repo_structure.files.
The baseline repo must have 7–9 failing tests when bootstrapped cleanly.
The test_runner command must use --tb=short not --tb=no.
```

---

## Validation before seeding

After Claude produces the JSON, run the signal audit checklist from Part 7 of the framework before seeding to the database:

- [ ] `min_reading_time_seconds: 300` is set
- [ ] Every failing test maps to a named symptom in the incident docs
- [ ] Diagnostic docs contain at least one specific ID, error text string, and .py function name
- [ ] Scenario explicitly asks for a written stakeholder summary
- [ ] Rubric weights are unequal and sum to 1.0
- [ ] Every rubric dimension has a `task_artifact` in `jd_to_signal_map`
- [ ] `must_not_cover` has at least 2 entries
- [ ] Bootstrap and test collection succeed locally before seeding

The canonical catalog must always contain exactly the tasks in `backend/tasks/`. The top-level `tasks/` directory (if it exists) is retired and must not be used.
